В Linux и подобных UNIX-подобных системах адресное пространство процесса организовано как виртуальное пространство, которое изолировано от других процессов и управляется ядром. Это позволяет каждому процессу работать с памятью, как будто она полностью принадлежит ему, несмотря на то что физическая память может быть разделена между несколькими процессами. [```1```](https://en.wikipedia.org/wiki/Virtual_address_space)[```2```](https://habr.com/ru/companies/timeweb/articles/804865/)

## Виртуальное адресное пространство

Виртуальное адресное пространство (VAS) — это набор виртуальных адресов, которые процесс может использовать для доступа к памяти. Оно делится на две основные части:
1. **Пространство пользователя** — область, с которой работают пользовательские приложения. Здесь располагаются код, данные, стек и куча процесса. [```2```](https://habr.com/ru/companies/timeweb/articles/804865/)[```12```](https://intuit.ru/studies/courses/641/497/lecture/11316)
2. **Пространство ядра** — зарезервировано для исполнения кода ядра, расширений ядра и большинства драйверов устройств. Доступ к нему из пользовательского пространства запрещён. [```2```](https://habr.com/ru/companies/timeweb/articles/804865/)

В 32-битных системах виртуальное адресное пространство обычно составляет 4 ГБ. В 64-битных системах теоретически оно может достигать 2⁶⁴ Б, но на практике для x86-процессоров не превышает 2⁴⁸ Б (256 ТБ). [```3```](https://www.aldpro.ru/professional/ALSE_Module_13/processes.html)

Память в виртуальном адресном пространстве разделена на блоки фиксированного размера — **страницы** (обычно 4 КБ). Ядро использует таблицы страниц для отображения виртуальных адресов на физические адреса в ОЗУ. Это позволяет:
* изолировать процессы друг от друга;
* эффективно управлять памятью;
* использовать механизм подкачки (swap), перемещая неактивные страницы на диск. [```1```](https://en.wikipedia.org/wiki/Virtual_address_space)[```2```](https://habr.com/ru/companies/timeweb/articles/804865/)

## Сегменты в адресном пространстве

Виртуальное адресное пространство процесса делится на несколько сегментов (областей) разного назначения:
* **Сегмент кода (text)** — содержит инструкции процессора. Отображается на соответствующие области исполняемого файла.
* **Сегмент инициализированных глобальных переменных (data)** — данные, которые изначально загружаются из исполняемого файла.
* **Сегмент неинициализированных глобальных переменных (BSS)** — изначально заполнен нулями. В исполняемом файле хранится только имя переменной и её размер, а выделение памяти и инициализация нулями происходит при запуске программы.
* **Сегмент стека (stack)** — используется для хранения фреймов стека при вызовах функций.
* **Куча (heap)** — область для динамического выделения памяти функциями `malloc()`, `new()` и т. п.
* **Дополнительные сегменты** для каждой разделяемой между процессами библиотеки (например, `libc`).
* **Области для отображения в память файлов** и разделяемой памяти (shared memory).
* В современном Linux в отдельные области памяти `vsyscall` и `vdso` вынесен часто вызываемый код ядра, не требующий системных полномочий. [```9```](https://parallel.uran.ru/book/export/html/536)

## Стек

**Стек** — это структура данных типа LIFO (last in, first out), используемая для хранения фреймов стека при вызовах функций. Каждый фрейм стека содержит:
* аргументы функции;
* адрес возврата;
* локальные данные.

В пользовательском пространстве стек растёт вниз — к более низким адресам. Размер пользовательского стека можно настроить с помощью команды `ulimit -s`. [```6```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fwww.baeldung.com%2Flinux%2Fkernel-stack-and-user-space-stack)

При вызове функции в стек добавляется новый фрейм, а при возврате из функции — удаляется. Переполнение стека может произойти при слишком глубокой рекурсии или выделении слишком большого объёма данных в стеке. [```11```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fstackoverflow.com%2Fquestions%2F79923%2Fwhat-and-where-are-the-stack-and-heap)

Стек ядра — отдельная область в пространстве ядра. К нему нет прямого доступа из пользовательского процесса. Он используется при системных вызовах и обработке прерываний. [```6```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fwww.baeldung.com%2Flinux%2Fkernel-stack-and-user-space-stack)

## Куча

**Куча** — это область памяти, используемая для динамического выделения памяти по запросу процесса. В отличие от стека, куча растёт вверх — к более высоким адресам. [```11```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fstackoverflow.com%2Fquestions%2F79923%2Fwhat-and-where-are-the-stack-and-heap)[```6```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fwww.baeldung.com%2Flinux%2Fkernel-stack-and-user-space-stack)

Память в куче выделяется с помощью системных вызовов (например, `brk()`, `sbrk()`) или функций уровня языка программирования (например, `malloc()` в C, `new` в C++). [```11```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fstackoverflow.com%2Fquestions%2F79923%2Fwhat-and-where-are-the-stack-and-heap)[```9```](https://parallel.uran.ru/book/export/html/536)

**Особенности кучи:**
* Управление кучей сложнее, чем стеком, так как требуется отслеживать свободные и занятые блоки памяти.
* Может возникать фрагментация — ситуация, когда между используемыми блоками остаются небольшие свободные блоки, что затрудняет выделение крупных блоков памяти. [```11```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Fstackoverflow.com%2Fquestions%2F79923%2Fwhat-and-where-are-the-stack-and-heap)
* При освобождении памяти возможны операции объединения смежных свободных блоков для уменьшения фрагментации.

В ядре Linux для управления памятью используются различные аллокаторы (например, `slab`, `kmalloc`), которые оптимизированы для конкретных сценариев использования. [```15```](https://habr.com/ru/articles/466831/)

## Дополнительные механизмы

* **Копирование при записи (Copy-on-write, COW)** — механизм, используемый при создании нового процесса с помощью `fork()`. Изначально родительский и дочерний процессы совместно используют физическую память. Копия страницы создаётся только при её изменении в одном из процессов. [```9```](https://parallel.uran.ru/book/export/html/536)
* **Рандомизация расположения адресного пространства (ASLR)** — механизм, усложняющий использование переполнений буфера за счёт случайного размещения стека и кучи в адресном пространстве. [```4```](https://tr-page.yandex.ru/translate?lang=en-ru&url=https%3A%2F%2Funix.stackexchange.com%2Fquestions%2F16595%2Flinux-kernel-logical-address-space-organisation)

Таким образом, система управления памятью в Linux обеспечивает изоляцию процессов, эффективное использование физической памяти и гибкость при работе с данными.
